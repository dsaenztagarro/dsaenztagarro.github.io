I built a couple of production MCP servers over the past year. Along the way I kept extracting the same patterns into a library — model definitions, CRUD tools, prompt strategies, form generation, OAuth flows. Eventually I realized I wasn't building servers anymore. I was building a framework.

That framework is [mcp-rune](https://mcp-rune.dev): **turn your data model into a complete MCP server.**

## The MCP scaling wall

The [Model Context Protocol](https://modelcontextprotocol.io/) is spreading fast — Claude Desktop, VS Code, Cursor and more all speak it now. But every framework I tried stops at the *transport* level:

1. **The official SDK** — protocol primitives. You wire every tool by hand.
2. **Protocol wrappers** (FastMCP, mcp-framework) — nicer registration, still one handler per tool.
3. **API converters** (Stainless, FastAPI-MCP) — generate tools from OpenAPI specs, with no intelligence layer.

If your server manages models — books, projects, activities, anything with CRUD — you write the same five handlers, the same validation, the same field docs, for every model. Ten models × five verbs = fifty hand-written tools, a picker that overflows, and an LLM that struggles to choose between them. Add prompts, forms and search and it's hundreds of lines of boilerplate per model.

This is the problem Rails solved for web apps in 2004. Convention over configuration. The model is the source of truth. Write it once, generate the rest.

## What mcp-rune is

mcp-rune is an opinionated, model-driven framework. You declare a model once, and you get tools, prompts, forms, docs and interactive UI for free.

```typescript
import { BaseModel } from '@mcp-rune/mcp-rune/core'
import type { AttributeDefinition } from '@mcp-rune/mcp-rune/core'

export class Book extends BaseModel {
  static override endpoint = 'books'
  static override attributes: Record<string, AttributeDefinition> = {
    title:  { type: 'string', required: true, description: 'Book title' },
    author: { type: 'string', required: true, description: 'Author name' },
    status: { type: 'enum', enumValues: ['unread', 'reading', 'completed'],
              default: 'unread' },
    rating: { type: 'integer', description: 'Rating from 1-5',
              validation: { min: 1, max: 5 } }
  }
}
```

Thirty lines of config. From that, the framework builds six things.

## The six pillars

- **Polymorphic CRUD** — a fixed set of eight generic tools (`list_models`, `find_model`, `create_model`, `update_model`, `delete_model`, `search_records`, `bulk_action_models`, …) serves *unlimited* models. The LLM passes the model name as a parameter; the tool count never grows, so tool selection stays accurate whether you register 10 models or 100.
- **Prompt strategies** — guided form filling that adapts to complexity: *stateless* (< 10 fields, docs only), *hybrid* (10–20 fields, docs + validation + summary), *stateful* (20+ fields, section-by-section progress). All derived from the model, so docs never drift.
- **Interactive MCP Apps** — schema-driven UI (forms, list views, record detail, search, autocomplete and multi-select pickers) rendered directly inside Claude Desktop and compatible hosts. Same model config that drives the tools; zero hand-written HTML.
- **Analysis Memories** — GraphRAG-powered semantic search and map-reduce aggregation over your records, backed by local pgvector.
- **Domain Intelligence** — encoded business rules, workflows and terminology so the server understands *your* domain, not just generic CRUD.
- **OAuth 2.1 + PKCE** — production-grade auth with audience-restricted tokens (RFCs 7636, 7591, 8414, 8707, 9728), on both stdio (local) and Streamable HTTP (remote, multi-user) transports.

Everyone else works at the tool/transport level. mcp-rune works at the **application** level. The full breakdown lives at [mcp-rune.dev](https://mcp-rune.dev).

## Proof: engineer-mcp

A framework is only as convincing as what it ships. So here is a real one.

[Engineer](https://engineer.dsaenz.dev) is a personal-development platform I run — *track your growth, build better habits.* It has a real data model: activities, roadmaps, domains, Bloom's-taxonomy levels, logged time. **engineer-mcp** is that data model turned into an MCP server with mcp-rune. No bespoke tool handlers — the model definition *is* the server.

Here's the source data, in Engineer's own UI — a handful of activities across Maths and Software Development, each with a kind, a domain, a Bloom level and logged time:

Now I open Claude Desktop and ask, plainly: *"Can you analyze all my activities and take conclusions about my activity sessions from different perspectives?"* Claude discovers the engineer-mcp tools, ingests the activities with a few polymorphic queries, and renders a rich visual report — an **Interactive MCP App**, not a wall of text:

Overview tiles, distribution donuts by kind and by Bloom's level, a per-session duration bar chart, the full activity list — all driven by the same `attributes` and field groups that define the model. Then Claude reasons over it across several perspectives and, crucially, surfaces the **data-quality gaps**: two activities with no duration set, every activity with a `null` domain and subdomain, nothing linked to a roadmap, zero sessions actually executed:

None of that required a single hand-written analysis tool. The semantic search and aggregation come from Analysis Memories; the gap detection falls out of the model's own schema (required fields, associations, enums). I described the domain once; mcp-rune did the rest.

## Who it's for

If you own a SaaS product, an API, an internal tool, or a data platform and you want it to be genuinely AI-accessible — not "we bolted on a chatbot," but *Claude can query, mutate, and reason over your real model* — mcp-rune gets you there without months of custom MCP plumbing.

## Try it

```bash
npm i @mcp-rune/mcp-rune
```

It runs on Node 20+ or Bun, ships full TypeScript declarations (autocomplete and type-checking out of the box), and is MIT-licensed. It's extracted from production servers that power real workflows daily — engineer-mcp is one of them — not a proof of concept.

Start at **[mcp-rune.dev](https://mcp-rune.dev)**. If you build something with it, I'd love to hear about it.
